# v2.0 — Major feature release (months 4–6)

Bigger bets. Each of these is XL and probably deserves its own sprint.

## GAP-0020 — Advanced player stats

- **target:** v2.0
- **status:** proposed
- **effort:** XL

### Problem

Competitors surface per-player deep stats: pass completion zones, xG contribution,
defensive actions, heatmaps. Scorepion currently only shows season aggregates.

### Why deferred

Depends on vendor decision (see GAP-0003) and a significant server-side data model
extension. Also a lot of design work.

### Proposed solution

- Vendor: StatsBomb or Opta (same decision as xG).
- New schema: `player_match_stats` table with ~40 columns per player-match.
- New tabs on player sheet: "This season", "This match" (when viewed from a match).
- Chart library upgrade — Victory Native or Skia for heatmaps.

### Acceptance criteria

- [ ] Top-5 league players' full stat sheet loads in ≤ 2s
- [ ] Heatmap renders for last-5-match window
- [ ] Sync job keeps data ≤ 24h stale

### Dependencies

- Vendor contract
- Skia/Victory Native upgrade path

## GAP-0021 — Shot maps & match heatmaps

- **target:** v2.0
- **status:** blocked
- **effort:** XL

### Problem

No visualisation of where shots came from, no pitch-level heat data. Match detail
screen hits a ceiling quickly for hardcore fans.

### Why deferred

Blocked on GAP-0020 data vendor, plus major Skia / React Native Reanimated work to get
60fps pitch rendering.

### Proposed solution

- Per-match `shots` table (position, outcome, xG).
- Skia canvas rendering shot markers on a pitch SVG.
- Filter by team / player / time window.

### Acceptance criteria

- [ ] Pinch-zoom pitch
- [ ] Shots tappable with detail popover
- [ ] Renders at 60fps on mid-tier Android (Pixel 6a baseline)

### Dependencies

- Vendor (GAP-0003, GAP-0020)
- Skia proof of concept

## GAP-0022 — Social layer v2

- **target:** v2.0
- **status:** proposed
- **effort:** XL

### Problem

Groups today are private leagues — invite-only, no discovery, no feed. Missing the
social loop that drives DAU in competing apps.

### Why deferred

Social design decisions (moderation, reporting, blocklist, spam) need a policy first.
Launching without them is a community-health risk.

### Proposed solution

- Public group discovery with "trending" ranking.
- Comment threads on predictions (opt-in per user).
- Report + block primitives.
- Moderation queue for admins.

### Acceptance criteria

- [ ] Policy doc written (what's allowed, escalation path)
- [ ] Report flow tested
- [ ] Blocklist honoured everywhere a user-id appears
- [ ] Moderator tooling deployed

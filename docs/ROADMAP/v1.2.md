## GAP-0010 — GDPR data export

- **target:** v1.2
- **status:** proposed
- **effort:** M

### Problem

EU users have the right to a copy of their personal data under GDPR Article 15.
Currently we only support account deletion (Article 17).

### Why deferred from v1

No EU marketing push for v1. We can respond to any manual request via a DB export until
the self-serve flow is built.

### Proposed solution

- `GET /api/account/export` returns a signed URL to a generated ZIP containing:
  - `profile.json` — user row (sanitised)
  - `predictions.csv` — all predictions with match metadata
  - `group_memberships.csv`
  - `push_tokens.csv`
  - `audit_log.csv` — logins, language changes, deletion events
- Generated asynchronously (job queue), email notification with one-time link (24h TTL).
- Settings screen: "Download my data" button.

### Acceptance criteria

- [ ] Export completes within 60s for an account with ≤1000 predictions
- [ ] Signed URL expires, cannot be reused
- [ ] ZIP passes manual review for completeness
- [ ] Documented in Privacy Policy

### Dependencies

- Object storage bucket (S3 / R2 / GCS) configured
- Email provider (already set up for auth)

## GAP-0011 — Parental consent (COPPA / under-13 flow)

- **target:** v1.2
- **status:** blocked
- **effort:** L

### Problem

Our onboarding age gate blocks under-13 users entirely. US COPPA and UK ICO rules allow
under-13 participation with verifiable parental consent — we're leaving that audience
on the table.

### Why deferred from v1

Verifiable parental consent is a legal minefield. Requires either credit-card micro-charge,
government ID, signed form, or third-party provider (Kids Web Services, PRIVO). Decision
needs counsel input.

### Proposed solution

To be scoped after legal review. Likely path: integrate a third-party consent provider
rather than build in-house. Under-13 accounts get a reduced feature set (no public
leaderboard, no groups with strangers, no push).

### Acceptance criteria

- [ ] Legal memo signed off by outside counsel
- [ ] Provider selected and integrated
- [ ] Under-13 feature matrix documented
- [ ] Privacy Policy updated with COPPA section

### Dependencies

- Legal review (external counsel)
- Decision: build vs. buy

## GAP-0012 — Stripe activation

- **target:** v1.2
- **status:** scoped
- **effort:** M

### Problem

Stripe is scaffolded but inert (`ENABLE_BILLING=false`). No revenue path for v1.

### Why deferred from v1

Premium tier pricing, feature matrix, and regional tax handling not locked. Better to
launch free-only and measure conversion intent before committing to a price point.

### Proposed solution

1. Decide pricing (likely $4.99/mo or $29.99/yr, two tiers: Free / Pro).
2. Define Pro feature gates in `src/lib/entitlements.ts`.
3. Flip `ENABLE_BILLING=true` in prod env.
4. Enable webhook endpoint behind Stripe's IP allowlist.
5. Apple / Google in-app purchase: store policies require IAP for digital goods on
   mobile. Add `react-native-iap` or `expo-in-app-purchases` — Stripe is web-only.
6. Tax: register for EU VAT OSS, US state sales tax nexus analysis.

### Acceptance criteria

- [ ] Pricing page live at `/pricing`
- [ ] Test purchase end-to-end on sandbox
- [ ] Webhook handling idempotent (verified by replay test)
- [ ] Subscription state reflected in user JWT claims within 5s of webhook
- [ ] Cancellation flow respected (access until period end)
- [ ] Play Console in-app product approved

### Dependencies

- Apple Developer + Google Play merchant account
- Tax registration
- `react-native-iap` evaluation

## GAP-0013 — Accessibility audit

- **target:** v1.2
- **status:** proposed
- **effort:** M

### Problem

We've been careful about `accessibilityRole` on pressables but haven't run a full audit
with VoiceOver/TalkBack. Prediction card, match detail, and league table are the riskiest.

### Why deferred from v1

Not a store rejection risk for v1. Google's pre-launch report will flag obvious issues
after upload — good enough to catch the worst offenders.

### Proposed solution

- Manual pass with TalkBack on the 10 most-used screens
- Axe DevTools on the web build
- Font-scaling test at 200% — ensure no clipped text
- Colour-contrast audit on Emerald brand tokens

### Acceptance criteria

- [ ] Every interactive element has a label
- [ ] All images have `accessibilityLabel`
- [ ] 200% font scale has no hidden controls
- [ ] WCAG AA contrast on all primary text
- [ ] Findings triaged into per-screen sub-tasks

## GAP-0001 — Dark mode

- **target:** v1.1
- **status:** proposed
- **effort:** M
- **owner:** (unassigned)
- **tracking:** —

### Problem

The app ships light-only. Football users open the app late at night during matches;
light theme on a dim bedroom screen is uncomfortable and hurts retention of evening users.

### Why deferred from v1

Emerald Minimalism design tokens only have light-palette values. Need a second palette +
audit of every hard-coded `#RRGGBB` in components.

### Proposed solution

Extend `src/theme/tokens.ts` with a dark palette, wrap the root in a theme provider that
reads `Appearance.getColorScheme()` with a user override in Settings (`system | light | dark`).
Run an audit script to flag any `color:"#..."` literals in component files.

### Acceptance criteria

- [ ] `Settings → Appearance` toggle: System / Light / Dark
- [ ] All 20+ screens render correctly in dark mode (screenshot snapshot test)
- [ ] Emerald brand colour adjusted for WCAG AA contrast on dark bg
- [ ] No regressions in language picker, onboarding, prediction card
- [ ] Preference persists across app restarts

### Dependencies

- Design: dark palette sign-off
- QA: visual regression harness

## GAP-0002 — Offline-mode for predictions

- **target:** v1.1
- **status:** proposed
- **effort:** L

### Problem

Users in commute / stadium / weak-signal contexts lose predictions when the submit call
times out. Currently shows a generic error toast; prediction is dropped.

### Why deferred from v1

Needs a local queue (MMKV or SQLite), a retry worker, and conflict handling when the
server has already accepted a duplicate. Not shippable as a weekend feature.

### Proposed solution

- Persist in-flight predictions to MMKV on submit failure.
- Background task retries with exponential backoff when connectivity returns
  (`@react-native-community/netinfo` listener).
- Server endpoint accepts an idempotency key so retries don't double-count.
- UI shows a "queued" badge on the prediction card until confirmed.

### Acceptance criteria

- [ ] Airplane-mode submit → reconnect → prediction appears in the cloud
- [ ] Duplicate submits with same idempotency key return the stored prediction, not an error
- [ ] Queued predictions survive app kill
- [ ] Queue depth observable via `/api/health`

### Dependencies

- Server: idempotency key column on `predictions` table
- Client: MMKV or SQLite dependency

## GAP-0003 — xG on match details

- **target:** v1.1
- **status:** blocked
- **effort:** M

### Problem

Competitors (FotMob, Sofascore) show expected goals on match detail pages. Without it,
Scorepion's match detail screen looks thinner than expected.

### Why deferred from v1

API-Football Pro does **not** include xG. Options: upgrade to Opta, StatsBomb, or a
mid-tier vendor like Football-Data.org. Decision needs commercial/legal review.

### Proposed solution

After vendor pick: add `xg_home`/`xg_away` to the `football_fixtures` table, new sync
job pulling per-match xG, new section on match detail between "Stats" and "Lineups".

### Acceptance criteria

- [ ] xG displayed for every finished Premier League match going back to 2023-24
- [ ] Live xG updates during in-play (if vendor supports)
- [ ] Missing-data fallback: hide the section rather than show "0.00"

### Dependencies

- Commercial: vendor contract signed
- Legal: review of vendor ToS for display rights

## GAP-0004 — Cross-device sync QA pass

- **target:** v1.1
- **status:** proposed
- **effort:** S

### Problem

Predictions, XP, streaks, and language preference sync should work across a user's
phone + second device. We have the infrastructure (JWT, server as source of truth) but
no end-to-end test proving it.

### Why deferred from v1

Didn't have time to run the scripted scenario manually. Unit tests cover the endpoints;
the concern is stale TanStack cache on device B after a write on device A.

### Proposed solution

Scripted manual test + automated Playwright-on-web version:

1. Log in on A, submit prediction.
2. Log in on B — prediction appears without pull-to-refresh within 60s (background refetch).
3. Change language on A — B picks it up next foreground.
4. Delete account on A — B is signed out on next API call.

### Acceptance criteria

- [ ] Manual run documented in `docs/OPERATIONS.md`
- [ ] Playwright spec in `e2e/cross-device.spec.ts` green
- [ ] Any stale-cache bugs filed as sub-gaps

### Dependencies

- None.

## GAP-0005 — Push notification icon + sound polish

- **target:** v1.1
- **status:** proposed
- **effort:** S

### Problem

Android currently uses Expo's default notification icon. Sound is the system default.
Brand feel is weak compared to Sofascore/OneFootball push.

### Why deferred from v1

Needs design assets: 96×96 monochrome white notification icon PNG + optional custom
sound. Not ready in time.

### Proposed solution

- Generate `notification-icon.png` (white-on-transparent, 96×96, no colour).
- Configure `expo-notifications` plugin in `app.json` with `icon` + `color`.
- Optional: `notification-sound.wav` bundled and referenced by channel.
- Android notification channel per type: `predictions`, `settlements`, `streaks`.

### Acceptance criteria

- [ ] Push on Android 13+ shows the brand emerald tinted icon
- [ ] Each notification type has its own channel (user can mute one without muting all)
- [ ] Channel names localised

### Dependencies

- Design: icon asset
- Already-done: FCM server key (see `docs/DEPLOYMENT.md`)
